# Автономная сдача лендинга: один линк → «готово»

> Операционная модель. Заказчик даёт **ссылку на макет** (+ папку, ТЗ) и один раз
> отвечает на пакет вводных вопросов. Дальше — БЕЗ него: оркестратор раскидывает
> работу субагентам (Sonnet/Haiku), ревизоры всё проверяют, и заказчику уходит
> ОДИН финальный отчёт. Цель — быстро, автономно, дёшево по токенам, масштабируемо.
> Поблочного ручного ревью заказчиком быть НЕ должно.

## Зафиксированная конфигурация (заказчик, 30.06)
- **Автономность:** до конца, ОДИН финальный отчёт. Спорное решаю сам, помечаю в отчёте. Заказчика по ходу не дёргать.
- **Тиринг — Sonnet-heavy:** почти вся верстка и ревизия — на **Sonnet**; **Haiku** — только совсем тривиальная механика; **Opus** — оркестрация + по-настоящему спорные решения. (Приоритет — качество первого прохода: меньше циклов починки = по факту меньше токенов и нет «нянченья».)
- **Workflow:** включаю **автоматически** для крупной задачи (лендинг целиком) — параллельный fan-out по блокам + ревизоры.

## Поток (5 фаз)

**0. Интейк** (Opus-оркестратор, дёшево):
- по ссылке вытянуть структуру: `get_metadata` → список фреймов/нод → **список блоков**;
- `get_design_context`/`get_screenshot` ключевых нод, свериться с reference-проектами
  (0066/0067) и `SECTION-GUIDE.md` на готовые паттерны;
- **собрать ВСЕ неясности и задать их заказчику ОДНИМ пакетом** (`AskUserQuestion`,
  ≤4 за раз, с рекомендованным вариантом первым). После этого заказчика не дёргать —
  спорное решаю сам и помечаю в финальном отчёте.

**1. План:** инвентарь блоков · ассеты · токены темы · какие блоки переиспользуют
друг друга (один компонент/файл, не клонировать) · что глобальное (в `COORDINATION`).

**2. Сборка — fan-out субагентами** (`Workflow` для детерминированного параллелизма):
один субагент на блок. **Модель по сложности (тиринг = экономия токенов):**
- **Haiku** — тривиальная механика: переименования, копирование паттерна, мелкие правки.
- **Sonnet** — основная верстка блока (pug+scss) по конвенциям. Это рабочая лошадь.
- **Opus** — только оркестрация + по-настоящему спорные дизайн-решения.
- `isolation: worktree`, если блоки параллельно мутируют общие файлы.

**3. Ревизия — adversarial, на каждый блок** (субагент-ревизор, Sonnet):
гоняет **Definition of Done из `BLOCK-DELIVERY-CHECKLIST.md` §2** —
скриншот-сверка с макетом на 3+ ширинах, видимость элементов (`naturalWidth>0`,
computed `fill/stroke/opacity/color`), нет оверфлоу. Нашёл расхождение → задача
обратно в починку (Sonnet). Цикл **build → review → fix**, пока чисто.

**4. Сборка целиком + сквозная проверка:** `npm run build` (exit 0, `validateHtml`),
оверфлоу страницы на всех ширинах, битые ссылки/картинки. → **ОДИН финальный отчёт**:
что сделано, где осознанные компромиссы, что (если что) требует решения заказчика.

## Токены: дёшево и быстро
- **Тиринг моделей** Haiku < Sonnet < Opus — дорогое только там, где реально нужно.
- **`Workflow`** для fan-out: параллельно, без болтовни, детерминированно.
- Не перечитывать файлы зря; батчить независимые вызовы; не дублировать ассеты;
  вопросы — одним пакетом в начале, не по ходу.

## Роли сессий
- **Оркестратор** (1 сессия): интейк → план → fan-out → финальный отчёт.
- **Рабочие/ревизоры** — субагенты внутри оркестратора ИЛИ боковые сессии по таблице
  `COORDINATION.md` (один блок — одна сессия, глобальные файлы — с пометкой).

## Что нужно от заказчика, чтобы запустить
1. Ссылка на макет (фрейм/страница целиком, не один блок).
2. Название папки проекта + ТЗ (`TZ/`), если есть.
3. Ответить на пакет интейк-вопросов **один раз**.
→ Дальше: «вернусь, когда готово».

## Связанные документы
- `BLOCK-DELIVERY-CHECKLIST.md` — Definition of Done + каталог граблей (этим гоняет ревизор).
- `SECTION-GUIDE.md` — конвенции верстки и «Поучения».
- `COORDINATION.md` — кто что делает, блокировки, общие конвенции.
